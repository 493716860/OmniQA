# 多账号会话隔离改造方案（Plan）

## Summary（摘要）

当前执行引擎 `DbTaskRunner` 使用单个 `requests.Session` 并在任务开始时回灌环境 Cookie、在每次请求后持久化环境 Cookie（`EnvironmentCookie`），导致**同一环境内多账号交替执行**时出现 Cookie 互相覆盖、链路串联错乱。  
你提出的“放弃环境 Cookie，改用 extractors 提取 token 并在 Header 使用 Authorization”能解决**Token 体系**下的大部分问题，但在现有架构里仍存在两类风险：

1. **Cookie 仍可能影响请求**：`requests.Session` 会自动携带 CookieJar（即使 Header 里不显式写 Cookie）；同任务多账号时 Cookie 依然会相互污染。
2. **运行时变量池是任务级单例**：若多个账号都提取同名变量（如 `${token}`），后者会覆盖前者。

为满足你确认的目标（**Cookie+Token 都要隔离；Cookie 仅任务内存；通过显式 `session_key` 指定会话**），本计划改造为：**多会话（multi-session）执行模型** —— 每个 `session_key` 维护独立的 `requests.Session` + 独立的变量池。

---

## Current State Analysis（现状分析：基于代码）

### 1) Cookie 为什么会串联

文件：`backend/api/services/db_runner.py`

- Runner 初始化时：
  - `self.session = requests.Session()`
  - `self.load_environment_cookies()`：从 DB 表 `EnvironmentCookie` 回灌 Cookie 到 `self.session.cookies`
- 每次请求后：
  - `save_session_cookies()` 会解析响应 `Set-Cookie` 并最终调用 `persist_environment_cookies()` 把 Cookie 写回 `EnvironmentCookie`

这使得 Cookie 成为“环境级共享状态”，多账号在同环境执行时天然会互相覆盖。

### 2) Token/运行时变量为什么会串联

文件：`backend/api/services/db_runner.py`

- `self.variables = DbVariablePool()` 是任务级单例
- `apply_extractors()` 会把 extractor 结果 `save_value(name, values[0])` 写入该单例池

因此同任务多账号如果 extractor 名称相同（如 `token`），后者会覆盖前者。

### 3) 现有“禁用 Cookie Header”不足以隔离 Cookie

文件：`backend/api/services/db_runner.py`

`prepare_headers()` 会过滤 `cookie`，避免用户在 header 里显式传 `Cookie`。  
但 `requests.Session` 仍会自动为匹配域名的请求附带 Cookie，因此**同 Session 内多账号切换仍会污染**。

## Proposed Changes（改造方案：文件级别的 What/Why/How）

> 目标：在不引入额外“执行环境”概念的前提下，通过 `session_key` 在同一 `TestTask` 内实现多账号/多角色会话隔离；Cookie 不落库，仅存在任务生命周期内；变量（含 token）按会话隔离。

### A. 后端数据模型：为 ApiCase/ScenarioStep 增加 `session_key`

**Why**  
让每个用例/步骤明确归属到某个“逻辑会话”，从而选择对应的 Session（CookieJar）与变量池。

**What / Files**
- `backend/api/models.py`
  - `ApiCase` 增加字段 `session_key`
  - `ScenarioStep` 增加字段 `session_key`
- `backend/api/migrations/xxxx_add_session_key_to_case_and_step.py`

**How（建议字段定义）**
- 类型：`CharField(max_length=64)`
- 默认值：`"default"`
- 允许空：`blank=True`

兼容性：历史数据自动落在 `default`，不需要额外数据修复即可执行。

---

### B. 执行引擎：从“单 Session + 环境 Cookie 持久化”升级为“多 Session + 任务内存态”

**Why**  
隔离 Cookie 的最可靠方式是隔离 CookieJar；隔离 Token/提取变量最可靠方式是隔离变量池。两者都应按 `session_key` 分片。

**What / Files**
- `backend/api/services/db_runner.py`

**How（核心结构调整）**

1. 在 `DbTaskRunner.__init__` 中引入映射：
   - `self.sessions: dict[str, requests.Session] = {}`
   - `self.variable_pools: dict[str, DbVariablePool] = {}`
   - `self.base_variables: DbVariablePool`（仅加载 ProjectVariable + EnvironmentVariable 的“基础变量池”）

2. 增加上下文切换方法（示例命名）：
   - `activate_session(session_key: str) -> None`
     - 若 key 不存在：创建 `requests.Session()` 与 `DbVariablePool()`（并把基础变量加载进去）
     - 设置 `self.session` / `self.variables` 指向该 key 的对象

3. 执行每个 case/step 前决定 session_key：
   - 对 ApiCase：取 `case.session_key or "default"`
   - 对 ScenarioStep：取 `step.session_key or "default"`
   - 在 `run_case()` / `run_step()` 最开始调用 `activate_session(session_key)`

4. **满足“Cookie 仅任务内存”**：移除环境 Cookie 的读取/写入
   - 删除或停用：
     - `load_environment_cookies()`
     - `persist_environment_cookies()`
     - `save_session_cookies()` 内对 `persist_environment_cookies()` 的调用
   - 清理相关 import：`EnvironmentCookie`、`create_cookie` 等

5. 变量/提取逻辑天然按会话隔离
   - `apply_extractors()` 不需要改签名；只要确保它调用时 `self.variables` 已指向当前会话变量池即可
   - `${token}`、`${CASE_001.data.id}` 等替换均在当前会话内完成

**行为说明（对使用者的心智模型）**
- `session_key` 相同：共享 Cookie + 共享运行时变量（token、extractors、cookie_header 等）
- `session_key` 不同：Cookie 与运行时变量绝对隔离
- Project/Environment 变量仍可作为“基础配置”被每个会话读取（例如 base_url、公共 header、公共参数等）

---

### C. 序列化与校验（可选增强，但建议做）

**Why**  
避免 `session_key` 被填入不可控字符；并对“跨会话依赖”给出更早、更清晰的错误提示。

**What / Files**
- `backend/api/serializers.py`

**How**
- 增加 `session_key` 字符校验：仅允许 `[A-Za-z0-9_-]`，超出则报错
- （可选）依赖一致性校验：
  - `ApiCase.dependencies` / `ScenarioStep.dependencies` 若依赖对象 `session_key` 不同，提示“依赖链路跨会话，可能取不到变量/登录态”

> 注：如果你们希望允许跨会话依赖，这里就只做 warning 级别的 TaskEvent，而不是 serializer 阻断。

---

### D. 前端：为用例与场景步骤显式配置 `session_key`

**Why**  
你选择“显式 session_key 字段”，需要在编辑器里可见、可编辑，且默认 `default`。

**What / Files**
- `frontend/src/views/CasesView.vue`
  - 用例表单 `form` 增加 `session_key`
  - UI 增加输入项（推荐放在标题/编号附近）
- `frontend/src/views/ScenariosView.vue`
  - 步骤表单 `stepForm` 增加 `session_key`
  - UI 增加输入项

**How（交互建议）**
- 文案：`会话 Key (session_key)`
- placeholder：`default / userA / userB`
- 提示：相同 key 的步骤共享登录态与运行时变量；不同 key 相互隔离

---

### E. Cookie 管理功能的产品一致性处理（必须落地一项）

**Why**  
当前产品有“环境 Cookie 管理器（CookiesView）”与后端 `EnvironmentCookieViewSet`，并在 README/页面文案中强调“登录后 Cookie 会保存到环境维度”。  
而本需求明确为“Cookie 仅任务内存”，两者语义冲突，需要处理，否则会误导用户。

**What / Files（视选择而定）**
- 前端：
  - `frontend/src/views/CookiesView.vue`
  - `frontend/src/layouts/MainLayout.vue`
  - `frontend/src/router/index.js`
- 后端：
  - `backend/api/urls.py`
  - `backend/api/views.py`（EnvironmentCookieViewSet）

**How（推荐决策）**
- 推荐：下线/隐藏 Cookie 管理入口，并在后端接口上返回明确提示（Deprecated / Not supported in task-memory cookie mode）

## Assumptions & Decisions（假设与已决策）

### 已决策（来自你的回答）
- 隔离范围：Cookie + Token 都要隔离
- Cookie 生命周期：仅任务内存（不写入 EnvironmentCookie）
- 会话选择方式：显式 `session_key` 字段（在 ApiCase / ScenarioStep 上）

### 本计划默认假设
- 多账号链路需要“同任务内交替执行且互不污染”，因此必须引入“多 requests.Session”
- 环境/项目变量用于公共配置，可被每个会话共享（不会包含账号级敏感态）

## Verification Steps（验证与验收）

### 1) 后端自动化验证（建议补充/新增测试）
文件：`backend/api/tests.py`

1. **不再持久化环境 Cookie**
   - 执行包含 `Set-Cookie` 的 case/step
   - 断言 `EnvironmentCookie` 表记录数不因执行而变化（或相关写入逻辑已移除）

2. **同一任务不同 session_key 的 Cookie 隔离**
   - `userA` 登录返回 cookieA；`userB` 登录返回 cookieB
   - 后续各自业务请求应仅携带各自 cookie（通过 mock 或抓取 request headers 验证）

3. **Token/提取变量隔离**
   - `userA` extractor 提取 `${token}=A`，`userB` extractor 提取 `${token}=B`
   - 业务请求 Header `Authorization: Bearer ${token}` 应分别为 A/B，不互串

### 2) 手工验收（前端+任务执行）
1. 创建两条登录用例：
   - LOGIN_A：`session_key=userA`，extractor 提取 `token`
   - LOGIN_B：`session_key=userB`，extractor 提取 `token`
2. 创建两条业务用例：
   - BIZ_A：`session_key=userA`，Header `Authorization: Bearer ${token}`
   - BIZ_B：`session_key=userB`，Header `Authorization: Bearer ${token}`
3. 放入同一个 TestPlan 执行：两条业务用例均成功且鉴权不串
4. 重复执行同 TestPlan：不应“继承上次执行的登录态”（因为 Cookie 不落库）
